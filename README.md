# TextAnalyzer — Make-автоматизация Python-проекта

## Зачем это сделано

Make выступает единой точкой входа для управления окружением, зависимостями и проверками качества кода. Каждый target — не просто alias, а воспроизводимое действие с явным окружением. Цель — упаковать хорошие практики разработки в автоматизированный процесс.

Этот проект демонстрирует поэтапное построение пайплайна: от создания виртуального окружения до композиции проверок в единую команду `make check`.

---

## О проекте: что делает TextAnalyzer

TextAnalyzer — консольное приложение для анализа текстовых данных. В качестве источника используется публичный текст с Project Gutenberg (роман «Гордость и предубеждение»).

### Функциональность

1. **Загрузка текста** — модуль `service.py` скачивает текст по URL через `requests` и возвращает его как строку.
2. **Подсчёт слов** — модуль `calc.py` вычисляет общее количество слов в тексте.
3. **Средняя длина слова** — вычисляется средняя длина слова по всему корпусу.
4. **Частотный анализ** — модуль `service.py` использует `numpy` для подсчёта частоты встречаемости каждого слова (`np.unique`).
5. **FastAPI-заготовка** — в `service.py` создан экземпляр `fastapi.FastAPI()`, демонстрируя готовность к развёртыванию в виде веб-сервиса.

### Результат работы

При запуске `make run-app` приложение выводит в консоль:

```
Running TextAnalyzer
Words: 124567, Avg length: 4.23
```

(Точные цифры зависят от версии текста на Gutenberg.)

### Тестовое покрытие

Проект покрыт unit-тестами на `pytest`. Тесты проверяют:
- **`calc.py`** — подсчёт слов и средней длины (включая граничные случаи: пустая строка, пустой список).
- **`service.py`** — корректность HTTP-запроса (с мокированием `requests.get`) и частотный анализ слов.
- **`example.py`** — форматирование выходной строки.

Мокирование внешних вызовов позволяет тестировать логику без реальных сетевых запросов, что делает тесты быстрыми и детерминированными.

## Ключевые решения и обоснование

### 1. Окружение: явные пути вместо `source activate`

**Проблема:** Make запускает каждую строку рецепта в новом shell, поэтому `source .venv/bin/activate` не сохраняется между командами.

**Решение:** Переменные с явными путями, для каждой операционной системы:

```makefile
ifeq ($(OS),Windows_NT)
    PYTHON := .venv/Scripts/python
    PIP := .venv/Scripts/pip
else
    PYTHON := .venv/bin/python
    PIP := .venv/bin/pip
endif
```

Это гарантирует, что все команды выполняются в проектном окружении, а не в системном. By-design: окружение создаётся внутри проекта (`.venv/`), зависимости устанавливаются туда.

---

### 2. Зависимости: импорты vs requirements

**Проблема:** Код импортирует пакеты, но они могут быть не указаны в `requirements.txt`.

**Решение:** `scripts/check_deps.py` парсит AST файлов в `src/` и `tests/`, извлекает top-level импорты и сравнивает с `requirements.txt` + `requirements-dev.txt`.

**Особенности:**
- Игнорируются модули стандартной библиотеки (явный список `stdlib`).
- Игнорируются локальные модули проекта (файлы из `src/`).
- Неоднозначные случаи маппинга (`sklearn` → `scikit-learn`) заданы вручную — осознанный компромисс.

**Target:** `make check-requirements`.

---

### 3. Типы: статический анализ до рантайма

**Проблема:** Ошибки типов в Python проходят молча до исполнения.

**Решение:** `mypy` для статической проверки. Target назван `typecheck`, а не `mypy`, чтобы инструмент можно было заменить без изменения интерфейса.

---

### 4. Стиль: разделение проверки и исправления

- **`make format`** — переписывает файлы in-place (`black`).
- **`make lint`** — только проверяет (`black --check`), без изменений.

`lint` безопасен для CI, `format` используется локально.

---

### 5. Цепочки: композиция вместо копипаста

```makefile
check: typecheck check-requirements lint test
```

Make строит граф зависимостей. Один высокоуровневый target собирает атомарные проверки. Добавить новый шаг — одна строка.

---

### 6. Тесты: детерминированность через моки

**Проблема:** Тесты с реальными HTTP-запросами нестабильны и медленны.

**Решение:** `unittest.mock` (stdlib) подменяет `requests.get`. Проверяются оба сценария — успех и ошибка — без сети.

**Target:** `make test` (pytest). Включён в `make check`, поэтому полный пайплайн запускает тесты автоматически.

## Использование

```bash
make venv               # создать окружение
make install            # установить зависимости
make run-app            # запустить приложение
make typecheck          # проверить типы
make check-requirements # проверить зависимости
make lint               # проверить стиль
make format             # исправить стиль
make test               # запустить тесты
make check              # все проверки разом
```

## Проверка работы

```bash
make check          # все проверки должны пройти
make run-app        # приложение должно запуститься
make test           # тесты должны быть зелёными
git status          # неотслеживаемых файлов быть не должно
```
